# SSH4Agent

SSH4Agent is a small command-line helper for letting coding agents operate a
remote SSH workspace through short local commands.

It is designed for shared GPU servers where you do not want to install or log in
to Codex on the shared remote account. The agent edits a local mirror under
`workspaces/`, then uses SSH and rsync to run commands or sync files on the
remote machine.

## Features

- Short commands for agent prompts, such as `remote status -w default`.
- Multiple remote configurations, such as `remote1`, `remote2`, and generated
  wrapper commands.
- Multiple workspace mappings between local directories and remote directories.
- Separate OpenSSH config per remote under `tool/configs/<remote>/`.
- Guardrails for existing local or remote content before first sync.
- Compact remote status, GPU status, log tailing, and tmux helpers.

## Repository Layout

```text
SSH4Agent/
  tool/
    tool4remote                 # Python CLI entrypoint
    setup.sh                    # Generates local config and wrappers
    configs/
      examples/                 # Safe example configs
  workspaces/
  PLANS.md
```

## Requirements

- Python 3.11+
- OpenSSH client
- rsync
- Optional on the remote host: `nvidia-smi`, `tmux`, `git`

## Quick Start

Run setup from the repository root:

```bash
cd /path/to/SSH4Agent
bash tool/setup.sh
```

This creates local private files:

```text
tool/configs/remote1/ssh_config
tool/configs/remote1/config.toml
tool/bin/remote1
tool/bin/remote
workspaces/default/
```

These generated files are ignored by git. Fill in your SSH and workspace
settings:

```bash
vim tool/configs/remote1/ssh_config
vim tool/configs/remote1/config.toml
```

Add wrappers to your shell PATH:

```bash
export PATH="/path/to/SSH4Agent/tool/bin:$PATH"
```

Verify:

```bash
remote status -w default
remote pwd
```

## SSH Config

Each remote has its own OpenSSH config. Example:

```sshconfig
Host gpu
    HostName GPU_SERVER_HOST_OR_IP
    User azureuser
    Port 22
    IdentityFile /absolute/path/to/private_key
    ServerAliveInterval 60
    IdentitiesOnly yes
```

Jump host example:

```sshconfig
Host jump
    HostName JUMP_HOST_OR_IP
    User root
    IdentityFile /absolute/path/to/jump_key

Host gpu
    HostName GPU_INTERNAL_HOST_OR_IP
    User azureuser
    Port 22
    IdentityFile /absolute/path/to/gpu_key
    ProxyCommand ssh -F /path/to/SSH4Agent/tool/configs/remote1/ssh_config -W %h:%p jump
```

The tool calls SSH with `ssh -F tool/configs/<remote>/ssh_config`, so you do not
need to copy these entries into `~/.ssh/config`.

## Workspace Config

Each `tool/configs/<remote>/config.toml` defines remotes and workspaces:

```toml
[defaults]
workspace = "default"

[remotes.remote1]
host = "gpu"
ssh_config = "ssh_config"
ssh_options = []

[workspaces.default]
remote = "remote1"
local_dir = "workspaces/default"
remote_dir = "/home/user/codex_workspace"
rsync_flags = ["-az", "--info=progress2"]
excludes = [
  ".git/",
  ".venv/",
  "__pycache__/",
  "*.pyc",
  "outputs/",
  "logs/",
  "checkpoints/",
  "wandb/",
  "runs/",
  "tmp/",
]
status_commands = []
```

`tool4remote` automatically loads all matching configs:

```text
tool/configs/*/config.toml
```

Use examples from:

```text
tool/configs/examples/
```

## Commands

Full CLI:

```bash
tool/tool4remote list
tool/tool4remote status -w default
tool/tool4remote pull -w default
tool/tool4remote push -w default
tool/tool4remote run -w default "bash scripts/run_exp.sh --dry-run"
tool/tool4remote tail -w default logs/latest.log -n 100
tool/tool4remote gpu -r remote1
tool/tool4remote tmux -w default new exp1
```

Short wrappers generated by `setup.sh`:

```bash
remote status -w default
remote pull -w default
remote push -w default
remote run -w default "bash scripts/run_exp.sh --dry-run"
remote tail -w default logs/latest.log -n 100
```

With multiple remotes:

```bash
remote1 status -w default
remote2 status -w other
```

If the first argument is not a tool subcommand, the wrapper runs it as a remote
shell command in the configured remote workspace:

```bash
remote pwd
remote cd ./
remote1 "python train.py --dry-run"
```

Each wrapper call is a separate SSH session. Shell state such as `cd` does not
persist between calls. Use:

```bash
remote run -w default "cd subdir && bash run.sh"
```

## Existing Content Guardrails

`pull` and `push` protect uninitialized workspaces:

- `pull` stops if the local workspace already contains files not initialized by
  SSH4Agent.
- `push` stops if the remote workspace already contains files and the local
  workspace has not been initialized by SSH4Agent.

After inspecting the contents and deciding it is safe:

```bash
remote pull -w default --force
remote push -w default --force
```

Use `--delete` only when the source side is the source of truth:

```bash
remote pull -w default --delete
remote push -w default --delete
```

## Recommended Agent Prompt

When starting an agent inside a local mirror:

```bash
cd /path/to/SSH4Agent/workspaces/default
codex
```

Use a compact instruction:

```text
Edit only this local workspace. To validate remotely, use:
remote push -w default
remote run -w default "COMMAND"
Use remote status and remote tail for compact logs. Do not ssh-cat large files.
```

## Security Notes

- Do not commit real SSH configs, private keys, local mirrors, logs, or generated
  wrappers.
- Keep private keys `chmod 600`.
- Prefer personal remote workspaces rather than shared project directories.
- Treat `--force` and `--delete` as destructive sync operations.

The root `.gitignore` is set up to keep private local state out of the repository
while preserving safe examples under `tool/configs/examples/`.
